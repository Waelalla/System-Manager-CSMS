📋 مراحل إضافة نوع جديد للشكاوى (Complaint‑Type) في نظام إدارة خدمة العملاء
ملاحظة: هذه الخطوات مخصصة للمستخدمين الذين يملكون صلاحية إدارة الأنواع (عادةً دور مدير أو مهندس صيانة). كل خطوة تتضمن ما يظهر للمستخدم في الواجهة، ما يحدث في الخلفية، وما يجب التحقق منه قبل الانتقال للخطوة التالية.

1️⃣ الوصول إلى واجهة إدارة أنواع الشكاوى
ما يُظهره UI	ما يُنفّذه الـ Backend	ملاحظات الصلاحية
من القائمة الجانبية → الإعدادات → إدارة أنواع الشكاوى	يحمّل من API GET /api/complaint‑types جميع الأنواع المُعرَّفة مسبقًا ويخزّنها في الـ store (Redux / React‑Query)	يجب أن يكون الـ JWT للمستخدم يُحتوي على صلاحية COMPLAINT_TYPE_MANAGE؛ وإلا تُعرَض رسالة “ليس لك صلاحية”.
تظهر قائمة جدول الأنواع مع زر “إضافة نوع جديد” في أعلى اليمين	لا يحدث شيء بعد؛ فقط إظهار زر الإضافة.	زر الإضافة مقفل إذا كان المستخدم viewer فقط.
2️⃣ فتح نموذج إضافة نوع الشكوى
UI	Backend	ملاحظات
النقر على زر “إضافة نوع جديد” يفتح Modal أو صفحة فرعية تحمل نموذجاً فارغاً.	لا يستدعي API حتى الآن (الـ modal يُظهر نموذجاً يُعبأ محلياً).	يفضَّل أن تكون الـ Modal قابلة للـ responsive (حجم الهاتف/التابليت).
العناصر الأولية في النموذج:
1️⃣ اسم النوع (حقل نص)
2️⃣ وصف مختصر (textarea – اختياري)
3️⃣ فئة (Dropdown – مثال: جودة المنتج، خدمة ما بعد البيع، فني صيانة…)
4️⃣ حالة التفعيل (Toggle: فعال / غير فعال)	لا يتم إرسال أي شيء إلى الخادم بعد.	اسم النوع يجب أن يكون فريدًا داخل قاعدة البيانات؛ سيتم التحقق من ذلك عند حفظ النموذج.
3️⃣ تعريف الحقول الديناميكية للنوع
الحاجة إلى حقل JSON يُخزن في جدول complaint_types باسم fields. كل عنصر في هذا المصفوفة يمثل حقلًا يُظهر للمستخدم عند إنشاء شكوى من هذا النوع.

خطوة UI	تفاصيل الحقول	منطق Backend
زر “إضافة حقل” داخل النموذج يضيف سطرًا جديدًا في جدول المجالات.	لكل حقل يطلب:
• اسم الحقل (نص، مثال: “المنتج”، “اللون”، “المقاس”)
• نوع الإدخال (Select):
  - text (نص قصير)
  - textarea (نص طويل)
  - number
  - date
  - select (قائمة منسدلة)
  - rating (نجوم 1‑5)
  - file (رفع صورة/مستند)
• قائمة الخيارات (لـ select فقط) – يكتب القيم مفصولة بفواصل
• هل الحقل إلزامي؟ (Checkbox)
• قيم افتراضية (اختياري)
• قواعد التحقق (Regex أو حد أقصى/الحد الأدنى للـ number)
• ترتيب الحقل (Drag‑&‑Drop أو رقم تسلسلي)	عندما يُحفظ النموذج، الـ frontend يُحوِّل مصفوفة الحقول إلى JSON (مثال أدناه) ويرسلها إلى الـ API POST /api/complaint‑types.
الخلفية تتحقق من:
1. ف uniqueness لاسم النوع داخل الـ table.
2. عدم وجود أسماء حقل مكررة داخل نفس النوع.
3. صحة نوع الإدخال (مثلاً لا يُسمح بـ file إذا كان الإعداد العام للملفات مُعطَّل).
4. قواعد التحقق (Regex) في حال وجودها تُخزَّن وتُستَخدم لاحقًا في الـ front‑end عند إنشاء الشكوى.
أثناء إضافة الحقول يمكن للمستخدم معاينة النموذج (Preview) لتأكد من الشكل النهائي.	لا يُرسل طلب للـ API أثناء المعاينة؛ كل شيء يبقى محليًا.	المعاينة تساعد على تجنّب الأخطاء قبل الضغط على “حفظ”.
4️⃣ حفظ النوع الجديد وإبلاغ المستخدم
UI	Backend	ما يحدث بعد الحفظ
زر “حفظ النوع” يُرسل طلب POST /api/complaint‑types مع الحمولة:
{ name, description, category, active, fields: [{name, type, required, ...}] }	1️⃣ التحقق (Joi/Zod)؛ في حال فشل يُرجِع 400 مع رسائل تفصيلية (مثلاً “اسم النوع موجود مسبقًا”).
2️⃣ إنشاء السجل في جدول complaint_types.
3️⃣ تحديث الـ cache (Redis أو in‑memory) إذا كان مُفعَّل لتقليل استعلامات الـ DB.
4️⃣ إرسال إشعار عبر WebSocket/Socket.io لجميع الجلسات المفتوحة لتحديث قوائم الأنواع فورًا.	1️⃣ ينعرض Toast بنجاح “تم إضافة نوع الشكوى بنجاح”.
2️⃣ يُغلق الـ Modal أو يُعيد توجيه المستخدم إلى قائمة الأنواع حيث يظهر النوع الجديد في أعلى القائمة.
3️⃣ يصبح النوع متاحًا في قائمة اختيار نوع الشكوى على صفحة إنشاء شكوى.
5️⃣ اختبار النوع في سيناريو إنشاء شكوى
ما يُجرى	كيف يُتحقق
المشتغل (موظف خدمة عملاء) يفتح صفحة إضافة شكوى → يختار النوع الجديد من القائمة.	يجب أن يتولد نموذج ديناميكي يحتوي على جميع الحقول التي عُرِّفت في الخطوة 3.
يملأ الحقول وفق القواعد (مثلاً حقل select يُظهر القيم المحددة، حقل rating يعرض نجوم).	الـ front‑end يستخدم الـ JSON المخزن في الـ DB لتطبيق validation الفوري (Yup/Zod).
يحفظ الشكوى.	الـ API POST /api/complaints يتحقق من أن القيم المرسلة تطابق تعريف الحقول (المطلوبة، نوع البيانات، القواعد).
إذا نجح الحفظ → يُظهر إشعار “تم إنشاء الشكوى”.	إذا فشل → تُرجَع رسائل خطأ توضح أي حقل لم يمر بالتحقق.
